Frequently Asked Questions Make sure to read

Q: What if I can't run the reproduction code?
A: Focus on documentation and source code analysis. Most determinations can be made without execution.

Q: What if I'm unfamiliar with a package?
A: Do basic research (documentation, README, package description) to understand its purpose. You don't need deep expertise - maintainers are the ultimate arbiters. Reflect any uncertainty in your confidence scores.

Q: Should I consider backwards compatibility?
A: Yes - if fixing the bug would break existing code, note this in your assessment of whether maintainers would welcome it.

Q: What if the bug is already reported?
A: Mark as a real bug if it is one, but note in "would maintainers welcome" that it's already known and provide the link(s) to existing reports. This helps avoid duplicate work.

Q: How deep should I investigate?
A: Spend more time on ambiguous cases. Clear bugs or non-bugs should be quick decisions. Aim for reasonable care for a good-faith contribution.

Q: What counts as a security issue?
A: Any bug that could be exploited to: crash systems, corrupt data, bypass access controls, leak information, or consume excessive resources.
